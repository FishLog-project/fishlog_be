# geo.md — 낚시 스팟 위치·주변 검색

> 낚시 스팟의 좌표 저장과 위치 기반(반경/지도) 검색 설계. 좌표 저장(시드)은 🚧 진행중, 주변 검색은 📋 계획 단계.

## 요구사항
- 낚시 스팟을 위경도 좌표로 저장.
- 사용자 위치/지도 영역 기준 **주변 스팟 검색**(반경 N km, 또는 지도 bounding box).
- 지도 표시·지오코딩은 외부 지도 API 연동 → `docs/external.md`.

## 스팟 데이터 출처 🚧
- **출처:** 공공데이터포털 **바다낚시지수 API(15142486)** → `docs/external.md` §1.
- **저장 대상 필드(불변):** 위치명(`name` ← `seafsPstnNm`), 위도(`lat`), 경도(`lot`) — API의 불변 정보. 여기에 서비스 운영값 `prohibit`(낚시 금지 여부) 추가(ERD v0.1 → `docs/spec.md`). 예보성(지수·날씨·물때·대상 어종) 필드는 저장하지 않음.
- **규모:** 현재 **고유 위치명 49개 = 스팟 49종류**(추후 추가 가능). 지도에 마커로 표시.
- **좌표 신뢰원:** API 응답의 `lat`/`lot`을 그대로 사용(별도 지오코딩·수동 보정 없음). 지명 문자열(오프셋 표기 포함)은 표시용일 뿐 좌표 산출에 쓰지 않음.
- **적재 방식:** 사전 수집(배치) 후 시드 적재. 좌표가 불변이므로 초기 1회(또는 스팟 추가 시)만 수집.

## 저장 방식 ✅ (확정: 단순 위경도 컬럼)
- **채택:** **단순 위경도 컬럼** — `lat`/`lot` FLOAT 컬럼(ERD v0.1). 반경은 Haversine(또는 `ST_Distance_Sphere`), 지도 영역은 `lat`/`lot` BETWEEN.
- 규모(49개)가 작아 공간 인덱스 이득이 없어 이 방식으로 시작. 스팟이 수천 개 이상으로 커지고 반경 검색이 핫패스가 되면 그때 `POINT` + `SPATIAL INDEX`(SRID 4326)로 이관 검토.
- (참고) MySQL 공간 타입 방식은 `POINT SRID 4326` + `SPATIAL INDEX`(R-tree, MBR 연산만 가속) + `ST_Distance_Sphere`/`MBRContains`. 축 순서(4326은 `lat lng`)·Hibernate Spatial 의존성 주의.

## 질의 패턴 (초안)
- 반경 검색: 중심 좌표 + 거리 → 스팟 목록(가까운 순 정렬).
- 영역 검색: 지도 4개 좌표(bbox) → 화면 내 스팟.
- 성능: 스팟 수 규모에 따라 공간 인덱스 필요 여부 결정.
